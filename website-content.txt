System Design Architecture System structure and structural invariants A protected transfer is an ERC-20 transfer routed through a deterministic escrow with predefined release and dispute resolution paths. For auditors and security reviewers: Security Model. System flow A transfer moves through well-defined states. Terminal states are irreversible. State Meaning Terminal PENDING Created and funded, awaiting action No DISPUTED Dispute raised, awaiting resolver No RELEASED Funds sent to recipient Yes REFUNDED Funds returned to sender Yes RESOLVED Resolver outcome enforced onchain Yes The module system Sew Protocol uses pluggable, immutable modules to define escrow behavior. The protocol snapshots active module addresses at creation, ensuring that historical agreements are never retroactively affected by governance updates. Release Strategy Controls authorization and timing for settlement (Immediate, Time-based, or Conditional). Resolution Module Manages per-dispute state, resolver registries, and binary outcome enforcement. Yield Module Tracks per-escrow positions and principal across external protocols (Aave V3). Snapshotted at Creation Module addresses are permanently stored in the escrow state. Governance changes only affect new escrows. Isolated Namespacing Module state is indexed by (vault, workflowId). A module failure in one escrow cannot affect another. Module safety invariants No Silent Loss Modules must return all funds No Fund Redirect Modules only send funds back to the specific calling escrow vault. No discretionary redirection. Architectural invariants Non-custodial Funds held by contract. No operator or module can redirect assets outside defined transitions. Isolated Each escrow is independent. Failure in one cannot propagate to another. No shared risk. Forward-only Upgrades apply only to new escrows. Active agreements remain bound to original rules. Isolation as a first principle Between escrows Self-contained units. No shared collateral, shared yield pool, or shared dispute state. Between modules Separated logic components. A bug in yield cannot affect dispute resolution or core state. Between governance and settlement Governance cannot modify active escrows. Settlement logic is immutable once created. Emergency Recovery Containment-first response with bounded guardian powers This page documents the protocol's emergency response capabilities and recovery procedures. Emergency Philosophy Sew Protocol is designed with a containment-first approach to emergencies. The goal is to limit damage and preserve user funds, not to provide comprehensive recovery for all failure scenarios. Emergency mechanisms are intentionally constrained. They can reduce risk but cannot extract funds, modify existing escrows, or override settlement outcomes. Guardian Pause The guardian role can pause the protocol in response to critical vulnerabilities. Pause is an emergency circuit-breaker designed to prevent mass exploitation while fixes are prepared. What pause affects: New escrow creation Dispute initiation Dispute escalation What remains available during pause: Fund release (for non-disputed escrows) Fund cancellation (for non-disputed escrows) Existing position unwinding (where supported) Guardian Constraints Guardian powers are bounded to prevent abuse: Maximum pause duration 7 days Maximum pause cycles (per 90-day window) 3 pauses Can re-pause while paused No Can unpause Can extract funds No Can modify existing escrows No After the maximum duration, the pause automatically expires. Governance must manually unpause the system after addressing the underlying issue. Module Disabling The guardian can disable specific modules (such as yield generation integrations) without pausing the entire protocol. When used: A vulnerability is discovered in a specific integration An external protocol (e.g., Aave) experiences issues A resolution module shows problematic behavior Effects of module disable: New escrows cannot use the disabled module Existing escrows using the module may be restricted Other protocol functions continue normally Position Unwinding When an integration fails or poses risk, the protocol supports unwinding positions back to escrow custody. This allows users to exit risky positions while preserving their underlying funds. Unwinding process: Guardian identifies affected positions Unwind function is called for affected escrows Funds return to escrow custody Users can then release or cancel normally Unwinding preserves the principal. Yield may be forfeited depending on the timing and nature of the emergency. Emergency Response Timeline Phase Action Responsible Detection Issue identified (internal monitoring, external report, or incident disclosure) Team, auditors, or community Assessment Evaluate severity and scope of vulnerability Security team Containment Guardian pauses protocol or disables affected module Guardian Fix Preparation Develop and test remediation Development team Governance Review Timelock queue for fix deployment (if applicable) Governance Recovery Deploy fix, unwind positions if needed, unpause system What Emergency Mechanisms Cannot Do It is important to understand the boundaries of emergency response: Cannot recover funds sent to wrong addresses: Standard transaction errors are irreversible Cannot reverse settlement outcomes: Released, refunded, or resolved escrows are final Cannot modify escrow rules: Per-escrow immutability is absolute Cannot bypass release conditions: Settlement paths are deterministic Cannot compensate for external failures: Issues in connected protocols (e.g., oracle failures) may not be recoverable Post-Incident Process After an emergency is resolved: Incident report is published documenting what happened Root cause analysis is shared with the community Lessons learned are incorporated into protocol improvements Any emergency actions taken are documented and justified Related Pages Protocol Limits: Detailed constraints on guardian and governance powers Governance: How protocol changes are coordinated Security Model: Overall security philosophy and properties How It Works: Understanding escrow lifecycle Last updated: February 2026

Faq
───────────────────────────────────────────────────────────────────────────────

Can governance seize my funds? No. Governance can only change parameters for future escrows. Once funds are in an escrow, the rules governing that escrow are fixed. Governance cannot modify, redirect, or seize funds from any active or completed escrow. What happens if I send funds to the wrong address? Ethereum transactions are irreversible. If you send funds to an incorrect address, those funds cannot be recovered by the protocol, governance, or any third party. Always verify addresses before confirming transactions. What happens during a protocol pause? When paused, new escrow creation and dispute escalation are blocked. However: existing escrows can still be released, cancelled, or resolved; funds in active escrows remain safe; and the pause automatically expires after 7 days maximum. Is dispute resolution decentralized at launch? No. At launch, the protocol uses a single trusted resolver model. Decentralized dispute resolution (multi-resolver escalation with appeal bonds and staking) will be activated through governance after testing and validation—estimated Q4 2026. What chains are supported? Sew Protocol is deployed on Base (Ethereum L2). Multi-L2 support is designed into the architecture and may be added in future updates. Is Sew a wallet or marketplace? No. Sew is infrastructure—a security layer for transfers. Wallets, marketplaces, and applications build on top of Sew to provide protected transfers. Everyday Wallet is one example interface built on the protocol. Are my funds at risk if an integration (like Aave) fails? Yield generation is optional and protected by exposure caps. If an external protocol fails, exposure is limited to the capped amount. Core escrow funds remain separate and can be settled regardless of yield module status. Can escrow rules be changed after creation? No. Each escrow captures its complete configuration at creation—release strategy, resolution module, yield settings, and timeouts. These are snapshotted and cannot be changed by anyone, including governance. What fees does Sew charge? Escrow creation: 1% at launch (2% max). Yield protocol fee: 30% of generated yield. Appeal bond fee: 0% at launch (30% max, governance-activated later). See the Fees page for details. How long does dispute resolution take? Dispute duration is configurable per escrow with a maximum of 90 days. Simple disputes under single-resolver mode may resolve in hours. Complex disputes using escalating resolution may take longer depending on the escalation path.

Security
───────────────────────────────────────────────────────────────────────────────

Security First Security Model Primary reference for auditors and security teams Sew Protocol is designed as a containment layer for protected transfers on Ethereum. We focus on narrowing scope, isolating failures, and ensuring funds always follow deterministic paths. Jump to: Threat Model Dispute Economics Upgrade Semantics Emergency Recovery Isolation Model Security philosophy Ethereum transactions are irreversible. Sew approaches security through three core principles: Containment Failures may occur. The protocol is designed to limit their impact to the affected escrow. Determinism Funds move according to predefined rules, not discretionary human judgment. Isolation Each escrow is treated as an independent agreement with no shared state risk. Core security properties Funds are held by smart contracts, not by an operator or intermediary. Every escrow defines its rules at creation. Settlement follows predefined state transitions. Each protected transfer is independent. If one fails, others are unaffected. Upgrades do not alter existing agreements. Historical agreements remain stable. Threat model Sew is designed to operate in an adversarial environment. Each threat category includes the attack surface and structural mitigation. T1 Sending to the wrong address, premature release, or misunderstanding escrow state. Sender can cancel while PENDING. Dispute pathway available if uncooperative. T2 Funds held in escrow until conditions are met. Neutral resolver decides outcome. T3 Bugs in integration modules, yield strategies, or third-party contracts. Per-escrow isolation limits blast radius. Module boundaries prevent propagation. T4 Malicious proposals or compromised admin keys. Governance cannot touch active escrows. Changes are forward-only with timelocks. Isolation model Isolation is the primary containment mechanism, operating at three levels simultaneously. Between escrows Independent agreements. No shared funds or shared dispute state. Failure in one cannot propagate. Between modules Release, resolution, and yield logic are isolated. A bug in yield cannot affect resolution. Economic incentives Resolvers are incentivized to behave honestly through economic mechanisms: bonds, appeals, and slashing. Resolver bonds Resolvers post collateral when deciding. If overturned on appeal, they lose the bond. Appeal escalation Disputed decisions escalate to higher-tier resolvers with larger bonds and senior review. Governance and upgrades Governance can approve new modules, but cannot Emergency response A guardian role can pause high-risk operations (dispute initiation) for up to 7 days. Settlement operations remain available. Pause automatically expires. Review the technical architecture. Deep dive into the contract structure, state machines, and modular integration patterns.
